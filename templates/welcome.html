Welcome to RPA UI Log Analyzer

This prototype applies RPA UI interaction patterns to help automation designers move beyond simple click/keystroke replay. Upload a UI event log (CSV) and the tool will infer human-intent activities, match them to recognised RPA patterns, and recommend the appropriate automation method for each — broken down by execution environment.

Recommendation Pipeline

The analysis runs through six sequential steps. Steps marked LLM send data to a language model; steps marked Rule-based use deterministic logic; Hybrid steps use LLM with a rule-based fallback.

1
Event Grouping Rule-based

Raw CSV rows are merged into activity groups. Two consecutive events are placed in the same group when they share at least one key attribute (app, application, webpage, url, or element_id). A change in the app / application attribute triggers a context-switch boundary — the current group ends and a new one begins, flagged as a context switch.

2
Activity Inference LLM

Each activity group is sent to an LLM (batched in groups of 5 for efficiency, parallelised across up to 5 threads). The LLM reads the event names and key attributes (tag_name, element_id, browser_url, workbook, etc.) and returns a structured JSON with: an activity name in verb + object format (e.g. "Write credentials into username field"), the best-matching RPA pattern name, whether a prerequisite Find step is needed (requires_find), the target element to find, a context-switch detection flag with from/to context labels, supporting evidence observations, and a confidence score.

3
Action / Object Extraction Hybrid

The LLM's activity name is parsed into an Action (first word) and an Object (remainder). A rule-based normaliser then maps both to the AOMC pattern vocabulary — for example, "click / press / tap" all normalise to Activate; "type / input / paste" normalise to Write; clicking a text-field target is reclassified as Focus. If the LLM provided a pattern field, that name is matched directly against the pattern library (bypassing normalisation). Object normalisation reduces most targets to Element, but detects Option (dropdowns) and Context (window/application switches) as special cases.

4
Pattern Matching Hybrid

The normalised action + object pair is looked up in the 13-pattern library. Primary path: if the LLM supplied a pattern name, the library is searched for an exact name match (case-insensitive) in the correct execution context. Fallback: if no LLM pattern name is present, the rule-normalised action/object pair is matched against each pattern's Action and Object fields. The first match wins. If no pattern matches, the recommendation is left without a pattern assignment.

5
Context Identification Rule-based

Each group's execution environment is identified by scanning event attributes in priority order:

1. Web — if any event carries HTML-specific attributes: xpath, tag_name, html_id, browser_url, webpage, url.
2. Desktop — if HTML attributes are absent but app/UI-hierarchy attributes exist: application, app, window_title, workbook, worksheet, control_type.
3. Screen (visual) — if neither HTML nor app attributes are found but coordinate attributes exist: x, y, mouse_x, mouse_y.
4. Unknown — if no distinguishing attributes are found.

6
Method Recommendation Rule-based

The matched pattern carries a Method field listing one implementation approach per execution environment, separated by " / " (e.g. "HTML DOM manipulation (Web) / UI Automation manipulation (Desktop) / Hardware simulation (Visual)"). The tool selects the slice whose context label matches the identified environment and strips the label, returning just the technique name. This is the final method recommendation.

Implicit Activities

Two types of activity are automatically inserted into the sequence — they are not directly observed in the log but are necessary for a realistic RPA bot design.

Prerequisite Find Element Hybrid

When an activity targets a specific UI element (button, input, dropdown, etc.) the bot must locate that element before it can interact with it. A "Find [element]" step is automatically inserted immediately before any such activity. The LLM decides whether a Find is needed (requires_find) and names the element (find_target). A keyword-based rule provides the fallback.

Fixed confidence: 0.85

Context Switch Hybrid

When the active application changes between consecutive groups, a "Switch context from A to B" step is inserted at the boundary. The LLM detects the switch and names the from/to environments. A rule-based fallback checks the app / application attribute directly. Page navigation within the same site is not treated as a context switch.

Fixed confidence: 0.90

RPA Pattern Library (13 patterns)

All activities are matched against the 13 UI interaction patterns below. Each pattern has a fixed Action and Object, one or more supported execution environments, and a category.

Extraction — read or locate data without modifying the UI
Find Element Read Element Observe
Modification — change or remove UI content
Write Element Delete Element Disable Element
Control — navigate, trigger, or manage UI state
Open Activate Hover Switch Context Scroll Focus Refresh

Method Computation by Execution Environment

The recommended automation method depends on both the matched pattern and the identified execution environment. The table below summarises the method assigned for each environment across pattern categories.

Environment Detection signal Extraction method Modification / Control method
Web XPath, tag_name, browser_url, html_id, webpage HTML DOM parsing HTML DOM manipulation
Desktop application, app, workbook, worksheet, window_title, control_type UI Automation (UIA) tree parsing UI Automation (UIA) manipulation
Screen x, y, mouse_x, mouse_y, coordinates Visual recognition Hardware simulation

Note: Delete Element does not support the Screen environment. Switch Context is Desktop-only. Hover on Web also supports hardware simulation as an alternative.
